SQL Injection: Dari Dasar Logika Hingga Eksploitasi Rantai Protokol

SQL Injection (SQLi) adalah titik di mana keamanan aplikasi web runtuh karena ketidakmampuan sistem membedakan antara **data** dan **instruksi**. Di artikel ini, kita akan membedah semuanya sampai ke akar terdalam.

I. Dasar: Apa yang Sebenarnya Terjadi?

Bayangkan aplikasi web adalah perantara antara Anda dan Database. Saat Anda memasukkan input, aplikasi menyusun kalimat (Query) untuk dikirim ke Database. Kerentanan terjadi saat input Anda bisa mengubah struktur "kalimat" tersebut.

-- Query yang diinginkan Dev:
SELECT * FROM users WHERE id = '1';

-- Jika input adalah: 1' OR '1'='1
SELECT * FROM users WHERE id = '1' OR '1'='1';

Hasilnya? Database tidak lagi mencari ID 1, tapi mengecek apakah 1=1 (yang selalu benar). Pintu terbuka tanpa kunci.

II. 5 Fase Eksploitasi (Metodologi Pentest)

FASE 01 Discovery & Error Analysis

Menguji parameter dengan karakter seperti ', ", atau ). Tujuannya memancing database mengeluarkan verbose error yang membocorkan jenis DB (MySQL, MSSQL, dll).

FASE 02 Column Enumeration

Menggunakan perintah ORDER BY untuk menentukan jumlah kolom dalam query asli. Ini wajib dilakukan sebelum teknik UNION bisa dijalankan.

FASE 03 Database Fingerprinting

Menjalankan fungsi spesifik (seperti version() di MySQL atau @@version di MSSQL) untuk mengetahui versi sistem demi mencari exploit yang relevan.

FASE 04 Data Extraction (Dumping)

Mengambil nama tabel, nama kolom, dan akhirnya seluruh data user/admin dari database information_schema.

FASE 05 Privilege Escalation

Tahap "tamat" di mana penyerang mencoba membaca file sistem atau menulis file shell (RCE) ke server menggunakan fungsi INTO OUTFILE.

III. Teknik Tingkat Tinggi: Blind & Time-Based

Ketika server tidak menampilkan error apa pun, kita menggunakan teknik Inferensial. Kita mengirimkan logika "benar/salah" dan mengukur responnya.

Time-Based Injection

Ini adalah teknik paling lambat tapi paling akurat. Kita menyuruh database tidur selama beberapa detik jika tebakan kita benar.

' AND (SELECT 1 FROM (SELECT(SLEEP(5)))a)--

Jika website butuh waktu 5 detik untuk loading, berarti instruksi kita berhasil dieksekusi.

IV. Solusi Akhir: Pertahanan 3 Layer

Layer 1: Prepared Statements (Wajib)
Menggunakan Parameterized Queries. Ini memisahkan query dari data di level protokol, sehingga input user tidak akan pernah bisa dianggap sebagai instruksi.

Layer 2: Principle of Least Privilege

Jangan pernah menjalankan aplikasi web dengan user database root atau DBA. Batasi hak akses hanya pada tabel yang diperlukan.

Layer 3: WAF (Web Application Firewall)

Gunakan layer tambahan seperti Cloudflare atau ModSecurity untuk mendeteksi dan memblokir payload SQLi sebelum menyentuh server Anda.

Kesimpulan

SQL Injection akan tetap ada selama developer masih melakukan penggabungan string (concatenation) dalam menyusun query. Pahami mekanismenya untuk membangun pertahanan yang tidak bisa ditembus.
